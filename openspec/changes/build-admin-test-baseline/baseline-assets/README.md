# 复杂管理端测试基线包

本目录提供一套可直接复用的测试基线模板，用于复杂管理端项目的测试分析、回归范围判断、用例设计、自动化评估与测试报告输出。

## 资产层级映射

| 文件 | 资产层级 | 主要职责 | 主要使用阶段 |
| --- | --- | --- | --- |
| `module-map-template.md` | 结构层 | 沉淀模块边界、关键对象、链路与依赖关系 | 首版全量测试基线构建 |
| `role-permission-matrix-template.md` | 结构层 | 沉淀角色、权限项、数据范围与流程限制 | 首版全量测试基线构建 |
| `module-rule-library-template.md` | 结构层 | 沉淀规则 ID、规则来源与规则驱动的下游追溯 | 首版全量测试基线构建 + 后续迭代差异分析 |
| `iteration-diff-analysis-template.md` | 变化层 | 记录变更来源、影响面、原行为/新行为、回归要求与覆盖状态 | 后续迭代差异分析 |
| `test-strategy-template.md` | 执行层 | 定义本次测试范围、工作模式、精准覆盖目标与执行策略 | 后续迭代差异分析 |
| `test-case-template.md` | 执行层 | 以 RULE/DIFF 为输入，沉淀可执行验证单元 | 后续迭代差异分析 |
| `automation-mapping-template.md` | 执行层 | 评估 TC/RULE/DIFF 的自动化价值与落地方式 | 后续迭代差异分析 |
| `test-report-template.md` | 执行层 | 输出 DIFF/RULE/TC/AUTO 结果、未覆盖项与残余风险 | 后续迭代差异分析 |

## 工作阶段边界

### 首版全量测试基线构建

适用场景：项目第一次建立测试基线，或现有测试资产无法支撑统一追溯时。

本阶段输出：
- 模块地图模板
- 角色权限矩阵模板
- 模块规则库模板
- 迭代差异分析模板
- 测试方案、测试用例、自动化映射、测试报告模板
- 统一编号规则、来源回指规则和精准覆盖验收规则

本阶段不输出：
- 具体业务模块内容
- 真实项目测试用例
- 自动化脚本、执行脚本、数据脚本

首版推荐步骤：
1. 先建立 `module-map-template.md` 与 `role-permission-matrix-template.md`
2. 再建立 `module-rule-library-template.md`，沉淀 `规则 ID`
3. 最后确认 `iteration-diff-analysis-template.md`、`test-strategy-template.md`、`test-case-template.md`、`automation-mapping-template.md`、`test-report-template.md` 的使用方式与字段约束

### 后续迭代差异分析

适用场景：已经存在首版基线，本次只需要分析某个版本、冲刺或热修复的变化。

本阶段原则：
- 复用已有 `模块 ID`、`角色 ID`、`权限项 ID`、`规则 ID`
- 以差异项为单位记录来源、影响面、原行为/新行为、旧路径回归要求、回归策略和覆盖状态
- 只有在结构层发生变化时，才显式更新模块、角色、权限或规则资产
- 若变更仅影响现有规则行为，不重建全量测试基线

## 什么时候更新规则库，什么时候只更新差异分析

| 场景 | 是否更新规则库 | 是否更新差异分析 | 说明 |
| --- | --- | --- | --- |
| 新增模块规则、状态机、校验逻辑、权限规则 | 是 | 是 | 需要新增或修改 `规则 ID`，并记录版本影响 |
| 原有规则行为变化，但规则对象仍存在 | 是 | 是 | 规则库更新原规则定义，差异分析记录本次版本变化 |
| 仅某个版本引用既有规则做回归，不改规则定义 | 否 | 是 | 直接引用已有 `规则 ID` |
| 仅新增一个测试版本或热修复条目，结构层完全不变 | 否 | 是 | 不更新结构层资产 |
| 角色、权限模型、菜单结构发生变化 | 是 | 是 | 同时更新结构层和变化层 |

## 统一编号与局部编号建议

| 对象 | 编号格式 | 示例 |
| --- | --- | --- |
| 模块 | `M-<domain>-<nn>` | `M-ORDER-01` |
| 角色 | `R-<nn>` | `R-03` |
| 权限项 | `P-<module>-<nn>` | `P-ORDER-02` |
| 规则 | `RULE-<module>-<nn>` | `RULE-ORDER-05` |
| 差异项 | `DIFF-<release>-<nn>` | `DIFF-2026R1-04` |
| 测试方案 | `PLAN-<release>` | `PLAN-2026R1` |
| 用例 | `TC-<module>-<nn>` | `TC-ORDER-17` |
| 自动化项 | `AUTO-<module>-<nn>` | `AUTO-ORDER-03` |
| 测试报告 | `REPORT-<release>` | `REPORT-2026R1` |

局部辅助编号建议：
- 关键链路：`FLOW-<nn>`
- 模块依赖：`DEP-<nn>`

## 来源回指模型

以下模板必须至少支持 `来源类型`、`来源 ID/回指`、`来源摘要` 三类字段：
- `module-rule-library-template.md`
- `iteration-diff-analysis-template.md`
- `test-case-template.md`
- `automation-mapping-template.md`
- `test-report-template.md`

以下模板在涉及结构层变更时，建议补充来源回指字段：
- `module-map-template.md`
- `role-permission-matrix-template.md`
- `test-strategy-template.md`

推荐的来源类型：
- `PRD`
- `Story`
- `Bug`
- `Config`
- `TechTask`
- `Compliance`
- `Other`

## 字段必填规则

| 模板 | 必填字段 | 可留空字段 | 留空条件 |
| --- | --- | --- | --- |
| 模块地图 | `模块 ID`、`业务目标`、`关键对象`、`上游依赖`、`下游依赖`、`核心链路` | `来源类型`、`来源 ID/回指`、`来源摘要`、`结构变更标记` | 首版建立时可留空来源字段 |
| 角色权限矩阵 | `角色 ID`、`模块 ID`、`权限项 ID`、`数据范围`、`流程限制` | `来源类型`、`来源 ID/回指`、`来源摘要`、`关联差异项 ID`、`关联用例 ID` | 权限模型未变化时可留空来源字段 |
| 规则库 | `规则 ID`、`模块 ID`、`规则名称`、`来源类型`、`来源 ID/回指`、`来源摘要`、`规则描述` | `关联自动化项 ID`、`报告 ID`、`报告条目`、`排除原因` | 仅当尚未进入执行阶段时可留空下游字段 |
| 差异分析 | `差异项 ID`、`来源类型`、`来源 ID/回指`、`来源摘要`、`影响规则 ID`、`原行为/旧路径`、`新行为/新路径`、`回归策略`、`覆盖状态` | `关联自动化项 ID`、`报告 ID`、`报告条目` | 尚未进入自动化或报告阶段时可留空 |
| 测试方案 | `方案 ID`、`工作模式`、`引用基线版本` | `来源类型`、`来源 ID/回指`、`来源摘要` | 首版阶段可留空来源字段 |
| 测试用例 | `用例 ID`、`关联方案 ID`、`来源类型`、`来源 ID/回指`、`模块 ID`、`规则 ID` 或 `差异项 ID` | `关联自动化项 ID`、`关联报告 ID`、`报告条目` | 尚未进入自动化或报告阶段时可留空 |
| 自动化映射 | `自动化项 ID`、`关联用例 ID`、`关联规则 ID` 或 `关联差异项 ID`、`决策结果` | `关联报告 ID`、`报告条目` | 尚未进入报告阶段时可留空 |
| 测试报告 | `报告 ID`、`引用基线版本`、`差异项 ID / 规则 ID / 用例 ID / 自动化项 ID` 中至少一个、`覆盖状态` | `来源摘要` | 若来源信息已在上游完整维护，可按引用方式保留摘要 |

## 追踪链要求

本基线至少要支持以下链路：
1. `来源 -> 规则`
2. `规则 -> 用例`
3. `用例 -> 自动化决策`
4. `自动化/用例 -> 报告结果`
5. `报告 -> 规则/差异项/来源` 反向回查

如果某一环不存在，也必须显式标记原因，例如：
- 未形成规则 ID
- 暂无自动化价值
- 本次版本仅手工验证
- 环境阻塞导致未执行
- 已接受风险，暂不覆盖

## 精准覆盖验收标准

以下要求必须能通过模板输出直接检查：
1. 每个纳入范围的差异项，必须有唯一 `差异项 ID`，并记录 `来源类型`、`来源 ID/回指`、`来源摘要`、`影响规则 ID`、`回归策略` 和 `覆盖状态`
2. 每个关键规则，必须有唯一 `规则 ID`，并能关联到至少一个 `用例 ID` 或一个明确的排除原因
3. 每个纳入范围的用例，必须记录其 `模块 ID`，并在适用时记录 `规则 ID`、`差异项 ID`、`来源 ID/回指`
4. 每个自动化候选，必须有明确决策结果：`自动化 / 暂缓 / 不自动化`，并说明其上游 `规则 ID` 或 `用例 ID`
5. 每份测试报告，必须能按 `差异项 ID / 规则 ID / 用例 ID / 自动化项 ID` 展示覆盖结果、未覆盖项或残余风险
6. 未覆盖或部分覆盖的项，必须记录 `原因`、`风险等级` 和 `后续动作`

## 使用顺序

1. 首次建立基线时，先填 `module-map-template.md`、`role-permission-matrix-template.md`、`module-rule-library-template.md`
2. 版本进入测试时，引用既有结构层资产并填写 `iteration-diff-analysis-template.md`
3. 用差异分析结果驱动 `test-strategy-template.md` 和 `test-case-template.md`
4. 对高价值场景填写 `automation-mapping-template.md`
5. 测试完成后输出 `test-report-template.md`

## 非目标

- 不生成具体业务测试内容
- 不生成自动化脚本或执行脚本
- 不替代真实需求文档、缺陷系统或测试管理平台
