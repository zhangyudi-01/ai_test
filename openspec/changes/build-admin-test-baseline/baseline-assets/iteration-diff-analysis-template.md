# 迭代差异分析模板

## 1. 使用边界

- 本模板用于后续“迭代差异分析”阶段
- 前提是已经存在首版测试基线，至少具备模块、角色、权限、规则编号体系
- 若本次版本引入新的模块、角色、权限模型或规则模型，请在差异项中显式标记“需更新基线资产”
- 若本次仅沿用既有结构做回归，不要重建模块地图或权限矩阵，直接引用既有 `模块 ID`、`权限项 ID`、`规则 ID`

## 4. 风险评估准则

| 风险等级 | 判定条件 | 建议动作 |
| --- | --- | --- |
| 高 | 核心链路、金额/权限/审批、跨系统、状态流转重构 | 必做定向回归，必要时补充全链路回归与自动化 |
| 中 | 局部功能调整、已有成熟路径上新增分支 | 做模块回归并核对上下游联动 |
| 低 | 文案、低风险配置、边缘页面改动 | 做冒烟或抽样验证 |

## 6. 精准覆盖检查

| 检查项 | 判定标准 | 结果 |
| --- | --- | --- |
| 差异项完整性 | 每个纳入范围的变化都有唯一 `差异项 ID` | `<通过/不通过>` |
| 来源回指完整性 | 每个差异项都填写 `来源类型`、`来源 ID/回指`、`来源摘要` | `<通过/不通过>` |
| 规则映射完整性 | 涉及规则变化的差异项都填写 `影响规则 ID` | `<通过/不通过>` |
| 旧路径回归要求完整性 | 变化项都明确是否需要回归旧路径以及原因 | `<通过/不通过>` |
| 覆盖状态完整性 | 每个差异项都填写 `覆盖状态`，未覆盖项有原因与后续动作 | `<通过/不通过>` |
| 下游资产映射 | 高风险差异项已映射到 `用例 ID` 或 `自动化项 ID` | `<通过/不通过>` |

## 7. 填写建议

- 每个差异项尽量独立，不要把多个不相关变化揉成一行
- 风险等级要有依据，不能只写主观判断
- 若本次变化只是沿用既有结构，请直接引用已有 `规则 ID`，不要新建重复规则
- 未覆盖项不能留空，必须写明原因和后续动作
