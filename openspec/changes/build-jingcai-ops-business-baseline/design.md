## Context

源需求为 `docs/requirements/竞猜赛事运营系统需求PRD文档.docx`，最新修订记录到 `V0.6 / 2025-07-02`。PRD 覆盖登录认证、风险管理、北单赛程管理、比赛监控、异常管理、变更管理、开奖管理、数据文件管理、基础信息管理、比赛数据维护、用户管理、操作日志和通知等业务域，同时混合引用 Modao 原型、钉钉文档、附件模板、外部数据源和文件传输规则。

当前仓库中的主 specs 仅覆盖测试基线相关能力：

- `openspec/specs/admin-test-asset-library/spec.md`
- `openspec/specs/admin-test-baseline-modeling/spec.md`
- `openspec/specs/admin-test-diff-analysis/spec.md`

本次变更需要单独建立业务需求基线，不得复用测试主 specs，也不允许把业务基线资产做成测试工作流资产。

当前约束如下：

- 必须基于完整 PRD 进行全量沉淀，而不是只做摘要。
- 无法在当前工作区内可靠读取的图片、原型截图和外部附件，必须显式标注“依赖外部原型补充”。
- 需要支持后续迭代差异分析，因此基线资产必须可追溯、可引用、可比较。
- 这次只构建需求工作流资产，不输出测试方案、测试用例、自动化脚本、测试报告和业务实现代码。

## Goals / Non-Goals

**Goals:**

- 建立一套可长期维护的业务需求基线资产库，覆盖模块、页面、角色权限、对象字段、规则、流程、外部依赖、文件流转、API 映射和待确认问题。
- 建立“首版全量基线构建”与“后续迭代差异分析”的边界，避免每次需求变化都重写整套基线。
- 建立统一编号体系、来源回指模型和跨资产追踪关系，支持从规则追到页面/对象/API，也支持从页面反查规则和来源。
- 建立可验证的覆盖契约，让评审者可以判断基线是否真的覆盖了 PRD 的核心业务语义。
- 对外部原型、附件和图片依赖做显式暴露，避免未来在缺失上下文时误用基线。

**Non-Goals:**

- 不生成任何测试工作流交付物。
- 不编写业务实现代码、数据库脚本、自动化脚本或接口实现。
- 不补写无法在当前仓库中可靠读取的原型细节。
- 不修改现有 `admin-test-*` 主 specs。

## Decisions

### 1. 业务基线资产单独放入 `business-baseline/`

采用 `openspec/changes/build-jingcai-ops-business-baseline/business-baseline/` 作为业务基线资产目录，根目录只保留 OpenSpec 核心 artifacts。

原因：

- 让 proposal / design / specs / tasks 与业务基线内容分层清晰。
- 避免与既有测试基线目录结构混淆。
- 方便后续 archive 后将业务资产整体迁移或继续维护。

备选方案：

- 复用历史 `baseline-assets/` 目录名：被拒绝，因为仓库里该命名已经承载测试资产语义。
- 全部堆在 change 根目录：被拒绝，因为文件会失去角色边界。

### 2. 采用“三层业务基线资产结构”

业务基线资产分为三层：

1. 结构层：`product-module-map.md`、`page-function-inventory.md`、`role-permission-list.md`
2. 语义层：`business-object-dictionary.md`、`business-rule-library.md`、`state-flow-and-key-processes.md`
3. 接口与治理层：`external-dependencies-and-file-flows.md`、`api-action-mapping.md`、`open-questions-and-ambiguities.md`、`iteration-diff-analysis-template.md`

原因：

- 结构层回答“系统有什么”。
- 语义层回答“系统按什么规则运作”。
- 接口与治理层回答“系统如何与外部交互，以及未来如何维护变化”。

备选方案：

- 单文档大总表：被拒绝，因为无法维护、无法做差异分析。

### 3. 采用统一编号体系

本次基线使用稳定 ID，而不是依赖 PRD 的自然语言标题。编号前缀如下：

- `BASE-`：基线版本
- `MOD-`：模块
- `PAGE-`：页面/功能单元
- `ROLE-`：角色
- `AUTH-`：权限包
- `OBJ-`：业务对象
- `FIELD-`：对象字段
- `RULE-GEN-`：通用规则
- `RULE-MOD-`：模块专属规则
- `RULE-XMOD-`：跨模块联动规则
- `STATE-`：状态
- `FLOW-`：流程
- `EXT-`：外部依赖
- `FILE-`：文件流转规则
- `ACT-`：API/操作动作
- `Q-`：待确认问题
- `DIFF-`：迭代差异项

原因：

- PRD 编号存在重复和跳号，例如多个 `(9)` 小节，不能直接当稳定标识。
- 后续 diff 分析需要引用稳定 ID，而不是依赖标题全文匹配。

备选方案：

- 仅使用章节名称作为标识：被拒绝，因为同名页面、弹窗和动作会冲突。

### 4. 采用统一来源回指模型

每个可追溯条目至少记录：

- `来源类型`：`PRD章节` / `PRD表格` / `原型链接` / `外部附件` / `接口表`
- `来源回指`：优先使用 `章节路径 + 标题 + 行段`，外部内容用链接序号或说明
- `来源说明`：需要时补充“依赖外部原型补充”或“PRD 未提供”

原因：

- 仅有自然语言“来源于 PRD”无法支持后续差异核查。
- 需要显式区分“正文可证实”与“外部原型待补充”。

备选方案：

- 只在文档头部记录统一来源：被拒绝，因为无法支持条目级追踪。

### 5. 定义两阶段工作模式

阶段 A：首版全量业务基线构建

- 输入：完整 PRD 文档及可访问的外部链接
- 输出：全量业务基线资产
- 更新策略：全部资产允许首次建立和结构化重写

阶段 B：后续迭代差异分析

- 输入：基线版本 + 新增/修改的需求来源
- 输出：差异分析文档，至少包含基线版本、受影响资产 ID、原行为、新行为、差异类型、影响面、是否回写基线
- 更新策略：差异未冻结前，优先更新 diff，不直接覆盖基线；需求确认后再回写基线

原因：

- 保留基线稳定性，避免“每次改需求就重写历史基线”。
- 让变更过程可审计。

备选方案：

- 直接覆盖基线：被拒绝，因为会丢失旧行为与差异上下文。

### 6. 定义业务基线覆盖契约

本次基线验收采用可验证输出，而不是抽象目标：

- `product-module-map.md` 覆盖 PRD 中全部一级业务模块和关键二级子域。
- `page-function-inventory.md` 覆盖 PRD 中全部显式页面、弹窗、详情页、列表页和待定页。
- `role-permission-list.md` 覆盖全部预设角色和系统权限表中的全部权限包。
- `business-object-dictionary.md` 覆盖核心业务对象，并为每个对象列出关键字段和约束。
- `business-rule-library.md` 覆盖通用规则、模块专属规则和跨模块联动规则，并带来源回指。
- `state-flow-and-key-processes.md` 覆盖显式状态映射和关键业务流程。
- `external-dependencies-and-file-flows.md` 覆盖数据源、认证器、外部原型、FTP/SFTP、导入导出和回执流转。
- `api-action-mapping.md` 覆盖 PRD 操作日志表中全部显式接口动作；PRD 缺失接口的动作必须标注 `PRD未提供`。
- `open-questions-and-ambiguities.md` 覆盖所有待定、缺失、依赖外部原型补充或接口未明的事项。

原因：

- 这是评审和后续维护的硬约束，不能依赖主观判断“差不多覆盖了”。

### 7. 所有缺失外部细节统一保守处理

如果某条需求依赖：

- 原型截图
- 钉钉附件
- 外部原型页面
- 数据源字段模型
- 示例模板细节

且当前工作区无法可靠读取，则该条需求不做推断性补写，统一记录为：

- `来源类型 = 原型链接 / 外部附件`
- `来源说明 = 依赖外部原型补充`

原因：

- 基线要稳定，不应把推断当事实沉淀进去。

## Risks / Trade-offs

- `[Docx 提取后版式丢失]` → 通过章节回指、保守表述和待确认问题清单控制误读风险。
- `[外部原型/附件不可直接读取]` → 在对应资产与疑问清单中显式标注“依赖外部原型补充”。
- `[PRD 编号不稳定且存在重复小节号]` → 使用统一 ID 替代原始编号。
- `[后续需求变化如果直接改基线，会丢失历史上下文]` → 用 `DIFF-*` 模板先承接变化，再在冻结后回写基线。
- `[权限、接口、导出模板等外部表格可能脱离 PRD 单独演进]` → 在外部依赖与疑问清单中记录引用关系，要求后续变更同步做差异分析。

## Migration Plan

本次变更不涉及运行时系统迁移。

文档迁移路径如下：

1. 在当前 change 下完成业务基线资产和 capability specs。
2. 评审通过后，按 OpenSpec 归档流程 archive change。
3. archive 时同步生成新的业务主 specs，不影响现有测试主 specs。
4. 后续若出现业务需求变更，以当前基线版本为输入创建 diff 资产或新 change。

回滚策略：

- 若变更未合入，直接移除该 change 目录。
- 若已合入但未 archive，回滚本 change 的文档提交即可。
- 若已 archive，则通过新的修订 change 进行纠偏，不直接篡改已归档历史。
