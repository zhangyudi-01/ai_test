## 两阶段工作模式

### 阶段 A：首版全量基线构建

适用场景：

- 首次接手系统
- PRD 尚未沉淀为可追溯资产
- 需要建立完整业务盘点

本阶段必须完成：

- 全量模块地图
- 全量页面/功能清单
- 全量角色权限清单
- 核心对象与字段词典
- 规则库、状态流转、关键流程
- 外部依赖、文件流转、API 映射
- 待确认问题清单

### 阶段 B：后续迭代差异分析

适用场景：

- 已存在稳定基线版本
- 新需求只影响局部模块、规则、对象、流程或接口

本阶段默认动作：

1. 先新增或更新差异分析文档，引用基线版本和受影响资产 ID
2. 对照“原行为 / 新行为 / 影响面 / 是否回写基线”
3. 需求冻结后，再回写对应基线资产

### 什么时候更新规则库，什么时候只更新差异分析

直接更新规则库的场景：

- 需求已确认落地，并且旧规则被正式替换
- 该规则已成为系统最新稳定业务事实

只更新差异分析、不立即改规则库的场景：

- 需求还在评审、待定、灰度或争议期
- 变更只在方案比选阶段，尚未形成正式业务结论
- 外部原型、附件或接口定义尚未补齐

## 来源回指规则

每个可追溯条目至少保留以下字段：

- `来源类型`：`PRD章节` / `PRD表格` / `原型链接` / `外部附件` / `接口表`
- `来源回指`：建议写到 `章节路径 + 标题 + 行段` 粒度
- `来源说明`：用于记录 `PRD未提供`、`依赖外部原型补充` 等说明

推荐写法示例：

- `PRD章节 | §基础信息管理 / 赛事管理 / (1)赛事列表 (L2149-L2166) | -`
- `原型链接 | 产品原型链接 #1 | 依赖外部原型补充`
- `接口表 | §操作日志 / 已实现操作日志 (L2565-L2652) | -`

## 必填字段与允许留空字段

### 所有清单类文档的必填字段

- `ID`
- `名称`
- `职责/描述`
- `来源类型`
- `来源回指`

### 有追踪关系的文档额外必填字段

- `上游/下游资产 ID`，或明确写 `-`
- `关联模块/页面/对象/规则/API`，或明确写 `-`

### 允许留空的字段

以下字段允许留空，但不能省略字段本身：

- `备注`
- `外部链接`
- `待确认问题ID`
- `来源说明`

留空原则：

- 已确认不存在时写 `-`
- 当前无法可靠读取时写 `依赖外部原型补充`
- PRD 未给出时写 `PRD未提供`

## 精准覆盖的可验证输出要求

满足以下条件，才算本基线“精准覆盖”达标：

1. `product-module-map.md` 覆盖 PRD 中所有一级业务模块。
2. `page-function-inventory.md` 覆盖 PRD 中所有显式页面、弹窗、详情页、待定页和关键动作入口。
3. `role-permission-list.md` 覆盖全部预设角色和系统权限总表。
4. `business-object-dictionary.md` 至少覆盖账号、角色、赛事、赛季、球队、比赛、奖期/批次、监控、异常、变更、开奖批次、提交记录、通知等核心对象。
5. `business-rule-library.md` 明确区分通用规则、模块专属规则、跨模块联动规则。
6. `api-action-mapping.md` 覆盖 PRD 操作日志表中全部显式接口动作，缺失接口必须显式标注。
7. `open-questions-and-ambiguities.md` 收录全部 `待定`、`未说明`、`依赖外部原型补充` 项。

## 外部原型补充策略

以下内容如果当前工作区无法可靠读取，不允许推断补写：

- 原型截图中的字段布局或默认值
- 钉钉附件中的排序规则、模板格式、示例图
- 外部数据源的字段模型
- 上传回执、变更单、公告表的具体版式

处理策略统一为：

- 先在对应资产里记录已知业务事实
- 缺失细节写入 `open-questions-and-ambiguities.md`
- 在 `来源说明` 中标记 `依赖外部原型补充`
