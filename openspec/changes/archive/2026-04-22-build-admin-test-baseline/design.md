## Context

当前仓库没有现成的业务系统上下文，因此这次变更的目标不是为某个具体管理端实例编写项目级测试内容，而是为“复杂管理端项目”定义一套可直接复用、可再裁剪、可追溯的测试基线包。该基线需要同时服务于测试分析、用例设计、自动化投入和版本回归决策，因此输出不能只停留在目录级说明，必须包含统一字段、编号规则、来源回指、引用关系和维护方式。

本次设计需要明确区分两个阶段：
- 首版“全量测试基线构建”：建立测试工作流资产、模板和追溯规则，形成统一输入面。
- 后续“迭代差异分析”：以既有基线为输入，记录本次变更来源、影响范围、覆盖决策和测试结果，而不是重新构建整套基线。

复杂管理端场景通常同时存在以下特征：多角色、多菜单、多状态流转、强规则约束、跨模块联动、批量处理与异步处理并存。测试基线若缺少结构化建模和明确的追溯规则，就会在版本推进中快速失真。因此设计重点是把结构建模、变更分析和执行资产打通，并把“精准覆盖”从抽象目标落到可检查的输出。

## Goals / Non-Goals

**Goals:**
- 提供一套面向复杂管理端项目的测试基线设计，覆盖模块地图、角色权限矩阵、模块规则库、迭代差异分析、测试方案、测试用例、自动化映射和测试报告模板。
- 为全部模板定义统一的标识规则、来源回指规则和下游引用规则，使模块、角色、权限、规则、差异项、用例、自动化项和报告结论之间可以双向追溯。
- 明确首版全量测试基线构建与后续迭代差异分析的边界，避免全量资产建设和单次版本回归混为一谈。
- 把“精准覆盖需求”固化为可验证的输出要求，使团队能够审计每个纳入范围的规则或差异项是否已经映射到方案、用例、自动化决策和报告结论。
- 让模板既能作为 OpenSpec 变更中的设计资产保存，也能被测试团队直接复制到项目文档或测试平台中继续完善。

**Non-Goals:**
- 不输出某一具体业务域的真实模块清单、真实角色配置、真实规则条目、真实测试数据、真实测试用例或真实报告结论。
- 不生成自动化脚本、测试执行脚本、数据工厂脚本或 CI/CD 集成配置。
- 不绑定某一种测试管理平台、自动化框架或报告工具的专有格式。
- 不在本次变更中实现自动生成器、执行引擎或与外部系统的集成。

## Decisions

### Decision 1: 用“两阶段工作模式”定义测试基线边界

本次设计将测试基线工作模式拆为两个阶段：
1. 首版全量测试基线构建：产出结构层、变化层、执行层模板，建立统一编号规则、来源回指规则和追溯关系。
2. 后续迭代差异分析：复用既有模块、角色、权限和规则编号，仅记录本次变更来源、受影响资产、回归策略、覆盖状态和结果回指。

阶段边界要求如下：
- 首版阶段不产出具体业务测试内容，只产出模板和工作流资产。
- 迭代阶段不重建全量结构资产，除非本次版本引入新的模块、角色、权限模型或规则模型变化。
- 若迭代阶段发现结构层变化，必须显式记录“需更新基线资产”的对象，而不是隐式修改。

Rationale:
- 复杂管理端的测试失控往往来自“结构资产没有沉淀”或“版本回归重复重建上下文”。
- 明确阶段边界后，测试团队能区分哪些是平台级资产建设，哪些是单版本回归活动。

Alternatives considered:
- 把首版基线建设和版本测试混在一套文档中：信息密度过高，后续难以维护。
- 每次版本都重新梳理模块和规则：短期看简单，长期会导致资产漂移和重复劳动。

### Decision 2: 用三层结构组织测试基线资产

测试基线分为三层：
1. 结构层：模块地图、角色权限矩阵、模块规则库
2. 变化层：迭代差异分析模板
3. 执行层：测试方案、测试用例、自动化映射、测试报告模板

Rationale:
- 复杂管理端的测试波动大多来自“结构不清”或“变化不清”，执行层问题通常是结果而不是起点。
- 先固定结构层和变化层，执行资产才能稳定维护，并为精准覆盖提供可审计上游输入。

Alternatives considered:
- 只输出测试方案和测试用例模板：无法解决回归范围判断、权限覆盖和规则沉淀问题。
- 以模块为中心平铺所有模板：容易重复记录角色、规则和差异信息，长期维护成本高。

### Decision 3: 用统一编号体系和来源回指模型打通上下游

所有模板共享统一标识符：
- 模块：`M-<domain>-<nn>`
- 角色：`R-<nn>`
- 权限项：`P-<module>-<nn>`
- 规则：`RULE-<module>-<nn>`
- 差异项：`DIFF-<release>-<nn>`
- 测试方案：`PLAN-<release>`
- 用例：`TC-<module>-<nn>`
- 自动化项：`AUTO-<module>-<nn>`
- 测试报告：`REPORT-<release>`

同时，规则库、差异分析、测试用例、自动化映射和测试报告必须至少支持以下来源回指字段：
- 来源类型：`PRD / Story / Bug / Config / TechTask / Compliance / Other`
- 来源 ID/回指：外部需求单号、缺陷单号、配置单号、技术任务单号或内部引用号
- 来源摘要：用于在脱离外部系统时仍能理解变更背景

Rationale:
- 统一编号能把模板从“文档集合”升级为“可追溯资产集合”。
- 来源回指能把测试资产与真实变更输入关联起来，避免只剩内部自引用。

Alternatives considered:
- 使用自由文本名称关联：可读性好，但在迭代和重命名后容易失效。
- 仅保留规则 ID 不保留来源回指：只能在内部串联，无法回查外部需求来源。

### Decision 4: 强制建立 `RULE -> TC -> AUTO -> REPORT` 追踪链

本次设计要求以下链路可顺向追踪并反向回查：
- `RULE -> TC`：测试用例必须能回指相关规则 ID；若不是规则驱动，也必须标记来源类型和来源 ID。
- `TC -> AUTO`：自动化映射必须能回指关联用例 ID，并明确自动化决策。
- `AUTO -> REPORT`：测试报告必须能引用自动化项 ID 或自动化分组结果。
- `REPORT -> RULE`：测试报告中的风险、未覆盖项或失败项必须能反查到规则 ID、差异项 ID 或来源回指。

Rationale:
- 用户要求的精准覆盖不是“有一些用例”，而是“能确认某条规则最终如何被验证、是否自动化、最终结论是什么”。
- 没有强制追踪链，报告只能给汇总数字，无法回答“哪些规则没有被有效覆盖”。

Alternatives considered:
- 只做 `DIFF -> CASE` 映射：适合简单功能回归，但不足以支撑规则型管理端场景。
- 只在报告中写问题总结：无法回查上游规则、用例和自动化决策。

### Decision 5: 将“精准覆盖需求”定义为可验证输出契约

精准覆盖的验收不使用抽象描述，而使用以下输出契约：
- 每个纳入范围的差异项必须在差异分析模板中有唯一记录，并标明来源类型、来源 ID、影响规则 ID、回归策略和覆盖状态。
- 每个关键规则必须在规则库中有唯一规则 ID，并能映射到至少一个测试用例或一个明确的排除原因。
- 每个纳入范围的测试用例必须标记其来源回指、模块 ID、以及适用的规则 ID/差异项 ID。
- 每个自动化候选必须记录是否自动化、为何自动化或不自动化、以及对应的用例 ID。
- 每份测试报告必须能按差异项 ID、规则 ID、用例 ID、自动化项 ID输出覆盖结果、未覆盖项和残余风险。

只有当上述输出都可直接在模板中检查到时，才视为满足“精准覆盖需求”。

Rationale:
- 抽象目标无法审计，输出契约可以被评审、复核和抽样检查。
- 管理端测试的真正难点在于覆盖边界是否清楚，而不是是否有文档存在。

Alternatives considered:
- 以“覆盖充分”作为描述性结论：缺乏统一口径，无法审查。
- 只统计用例数量或通过率：无法说明规则、差异和自动化决策是否真正闭环。

### Decision 6: 模板默认使用 Markdown 表格 + 补充说明块

所有交付物默认以 Markdown 保存，主结构采用表格，复杂说明使用补充段落或列表。

Rationale:
- Markdown 适合版本管理、代码评审和 OpenSpec 变更归档。
- 复杂管理端测试基线需要频繁 diff，文本化格式比二进制表格更适合审阅。

Alternatives considered:
- 直接输出 Excel 模板：录入效率高，但不适合在变更中审查和追踪。
- 全部使用纯段落模板：灵活，但难以形成结构化对比和自动化校验。

### Decision 7: 交付物作为独立模板文件存放在变更目录中

除 proposal/design/tasks/specs 外，本次变更还将输出独立模板文件，放在 `baseline-assets/` 目录下，供团队直接复用。

Rationale:
- 用户明确要求输出模板本体，而不是只在设计文档里描述模板。
- 独立文件可以直接复制到项目知识库、测试平台或仓库文档目录中使用。

Alternatives considered:
- 只在 design.md 中嵌入模板片段：阅读连续性好，但复用成本高。
- 仅在 tasks.md 中列出待产出模板：不能满足当前“输出基线”的目标。

## Risks / Trade-offs

- [风险] 追溯字段增加后，模板填写成本上升  
  → Mitigation: 把强制字段收敛为规则 ID、来源类型、来源 ID、上下游引用这几类核心键，避免过度设计。

- [风险] Markdown 表格在字段较多时可读性下降  
  → Mitigation: 将高维信息拆为主表 + 补充说明，避免单表承载全部维度。

- [风险] 没有具体项目上下文时，模板只能定义字段和工作流，不能提供真实样例内容  
  → Mitigation: 明确本次交付定位为“测试工作流资产与模板”，而非某一项目的真实测试材料。

- [风险] 如果团队不维护统一编号，追踪链会在后续版本中断裂  
  → Mitigation: 在 README、模板字段和验收标准中重复声明编号与回指要求，并通过评审抽样验证。

## Migration Plan

1. 在 OpenSpec 变更中建立 proposal、specs、design 和 tasks，固化基线设计意图、边界和追溯要求。
2. 在变更目录下生成 `baseline-assets/` 模板文件，形成可直接复用的测试基线包。
3. 在首版全量测试基线构建时，仅建立结构层、变化层、执行层模板与统一规则，不填充具体业务测试内容。
4. 在后续具体项目或具体版本中，先建立模块、角色、规则编号，再按版本填写差异分析、测试方案、用例、自动化映射和测试报告。
5. 根据试点反馈收敛字段，必要时迭代 spec 与模板结构。

## Open Questions

- 目标管理端更偏向 B2B 运营平台、内部审批平台，还是多租户 SaaS 控制台；这会影响模板示例的粒度和常见风险项。
- 团队后续准备落地到 Excel、XMind、TestRail、禅道或自研平台中的哪一种，以决定是否需要派生格式。
- 报告在实际落地中更偏向版本级报告还是专项报告，这会影响报告模板中对自动化项与用例结果的汇总方式。
