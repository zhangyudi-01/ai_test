## Why

复杂管理端项目通常具有多模块、跨角色、强配置、强流程依赖的特点，但测试输入分散、覆盖边界不稳定、迭代影响面难以快速识别，导致测试计划、用例设计和自动化投入缺少统一基线。现在建立一套可复用的测试基线资产，并明确“首版全量测试基线构建”与“后续迭代差异分析”的边界，可以把需求理解、风险识别、覆盖设计和回归决策前置，降低后续版本迭代的测试波动与沟通成本。

## What Changes

- 建立复杂管理端项目测试基线的统一输出结构，覆盖模块地图、角色权限矩阵、模块规则库模板、迭代差异分析模板、测试方案、测试用例、自动化映射和测试报告模板。
- 明确首版“全量测试基线构建”和后续“迭代差异分析”的边界：首版只构建测试工作流资产、模板、编号规则和追溯规则；后续迭代以既有基线为输入，仅对变更项、受影响规则和下游测试资产做差异分析与回归映射，除非基础结构发生变化，不重建全量基线。
- 定义面向管理端场景的模块拆解标准，明确主流程、关键对象、依赖关系、边界条件和跨模块联动点。
- 定义角色与权限维度的测试建模方式，使菜单、按钮、数据权限、流程权限和操作约束能够映射到测试覆盖。
- 为模块规则库、迭代差异分析模板、测试用例模板、自动化映射模板和测试报告模板增加统一追溯字段，至少支持规则 ID、来源类型、来源 ID/回指、以及下游资产引用。
- 将“精准覆盖需求”细化为可验证的输出要求，使规则、差异项、用例、自动化决策和报告结论之间可以形成闭环追踪链。

## Acceptance Criteria

- 首版全量测试基线构建必须输出完整的测试工作流资产包，至少包含模块地图、角色权限矩阵、模块规则库、迭代差异分析模板、测试方案、测试用例、自动化映射、测试报告模板以及统一编号与追溯说明。
- 基线文档必须明确两种工作模式的边界：首版全量测试基线构建用于建立结构层资产与追溯规则；后续迭代差异分析用于基于已有结构资产记录变更来源、影响面、回归策略和覆盖状态。
- 模块规则库、迭代差异分析模板、测试用例模板、自动化映射模板和测试报告模板必须都能记录规则 ID 与来源回指，且至少能形成 `RULE -> TC -> AUTO -> REPORT` 的顺向追踪，以及从报告反向回查到规则与来源项。
- 精准覆盖的验收必须可通过模板输出直接验证：每个纳入范围的差异项或关键规则，必须对应到测试方案中的覆盖主题、至少一个测试用例或明确的排除原因、自动化决策状态，以及测试报告中的结果或残余风险记录。
- 非覆盖项不能被隐式忽略，必须在差异分析、测试方案或测试报告中给出未覆盖原因、风险等级和后续动作。

## Non-Goals

- 不生成任何具体业务域的真实模块、真实角色、真实规则、真实测试数据或真实测试结论。
- 不编写或生成自动化脚本、测试执行脚本、数据准备脚本或 CI/CD 集成配置。
- 不绑定某一种测试管理平台、自动化框架或报告工具的专有格式。
- 不在本次变更中产出某一具体版本的业务测试内容，而只构建可复用的测试工作流资产与模板。

## Capabilities

### New Capabilities
- `admin-test-baseline-modeling`: 定义复杂管理端项目的模块地图、角色权限矩阵和模块规则库模板，作为测试分析的基础建模能力。
- `admin-test-diff-analysis`: 定义管理端迭代差异分析模板与风险识别规则，用于版本变更评估、回归范围圈定和测试优先级排序。
- `admin-test-asset-library`: 定义测试方案、测试用例、自动化映射和测试报告模板，并约束这些资产的结构、一致性与追溯关系。

### Modified Capabilities

None.

## Impact

- Affected systems: OpenSpec change artifacts under `openspec/changes/build-admin-test-baseline/`
- Affected process: initial baseline setup, change impact analysis, test planning, case design, automation selection, coverage audit, and reporting for complex admin projects
- Dependencies: OpenSpec spec-driven workflow only; no runtime code or external service changes
