# 自动化映射模板

## 1. 使用边界

- 首版全量测试基线构建：本模板用于定义自动化决策字段和评估方式，不生成脚本
- 后续迭代差异分析：本模板必须承接 `TC` 以及相关 `RULE/DIFF`，并能把结果回指到 `REPORT`

## 3. 决策参考

| 组合情况 | 建议 |
| --- | --- |
| 高价值 + 高频回归 + 高稳定性 | 优先纳入自动化 |
| 高价值 + 低稳定性 | 优先优化可测性，再决定是否自动化 |
| 低价值 + 高成本 | 通常保持手工覆盖 |
| 强依赖外部系统 + 断言困难 | 评估降层自动化或半自动验证 |

## 4. 分层建议

| 层次 | 适用场景 |
| --- | --- |
| UI | 页面流程、按钮显隐、端到端关键链路 |
| API | 权限校验、数据规则、稳定回归接口 |
| Service/Domain | 规则计算、状态流转、纯逻辑校验 |
| Job/Async | 定时任务、消息消费、异步补偿 |

## 5. 追溯要求

- 每个自动化项至少回指一个 `关联用例 ID`
- 若自动化直接来源于规则或版本差异，建议同时填写 `关联规则 ID` 与 `关联差异项 ID`
- 若决定“不自动化”或“暂缓”，必须填写 `未自动化原因`
- 若自动化结果需要进入测试报告，填写 `关联报告 ID` 与 `报告条目`

## 6. 填写建议

- 不要把“能写脚本”直接等同于“值得自动化”
- 至少同时评估业务价值和实现可行性
- 若一个场景已在低层覆盖充分，可只保留少量 UI 冒烟
