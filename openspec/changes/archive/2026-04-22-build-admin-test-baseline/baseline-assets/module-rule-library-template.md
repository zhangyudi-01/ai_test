# 模块规则库模板

## 1. 使用边界

- 首版全量测试基线构建：建立规则分类、`规则 ID`、来源回指规则和规则驱动的基础追踪关系
- 后续迭代差异分析：仅补充或更新本次受影响规则及其下游追踪关系，不重建全部规则库

## 2. 规则驱动原则

- 规则库是 `RULE -> TC -> AUTO -> REPORT` 追踪链的起点
- 规则描述优先表达“系统必须如何表现”，而不是记录零散测试点
- 若版本变化只复用既有规则回归，不新增 `规则 ID`
- 若规则定义、触发条件、状态机或副作用发生变化，必须更新本模板

## 4. 规则分类建议

- 输入校验
- 数据计算
- 状态流转
- 权限控制
- 配置驱动
- 异步处理
- 通知/消息
- 审计/日志
- 异常补偿

## 6. 覆盖状态说明

| 覆盖状态 | 含义 |
| --- | --- |
| 待映射 | 规则已建立，但尚未映射到具体执行资产 |
| 已覆盖 | 已有用例与结果链路，且当前版本已验证 |
| 部分覆盖 | 已有部分验证，但仍存在未覆盖路径 |
| 未覆盖 | 本次版本未对该规则建立有效验证 |
| 已排除 | 有明确排除原因并已记录接受方式 |

## 7. 什么时候更新规则库

- 新增规则、新增状态流转、新增配置驱动逻辑时更新
- 既有规则触发条件、成功结果、失败结果或副作用发生变化时更新
- 若仅在本次版本中引用既有规则回归，而规则定义不变，可只在差异分析和执行层模板中引用 `规则 ID`

## 8. 填写建议

- 规则描述尽量写成“当……时，系统必须……”的形式
- 一个规则只表达一个核心行为，复杂规则可以拆成主规则与子规则
- 有异常路径时，不要只写“提示错误”，需要明确错误表现与后续状态
- 首版基线阶段允许下游字段留空，但 `规则 ID`、`来源类型`、`来源 ID/回指`、`来源摘要` 不应缺失
