| 4 | 系统机制 | 空推 | 验证当无数据时，Kafka **不产生推送消息**，但 **Redis 时间持续更新** | 确保 Redis 游标持续滚动 |
|  |  | 批处理 | 验证单次查询超 500 条时，是否正确拆分为多个 Kafka 消息（如 500+1） | 验证数组最大长度 500<br>**构造 501 条符合条件的数据，验证批处理**<br>**预期**：收到 2 条消息，一条 size=500，一条 size=1。<br>Redis：必须等 2 条都推完才更新时间。 |

|  |  |  | **第一轮推送**：<br>*   预期：只推送 \[当天00:00:00 ~ 00:00:01\] 这一秒内的数据（通常为空或极少）。<br>    <br>*   Redis：更新为 当天00:00:01 |  |
|  |  |  | **第二轮推送** ：<br>*   预期：恢复正常逻辑，抓取 \[当天00:00:01 ~ 当前时间-Buffer\] 的全量数据。<br>    <br>*   验证：第二轮是否成功抓取了今天产生的所有积压数据。 |  |
|  |  | Topic **中断时间独立计算** | **验证每个Topic中断时间独立计算**<br>1. 修改 `bd_game_draw` 的 Redis 时间满足跨天>1h，但保持 `bd_ticket_with_win_amount` 时间正常。<br>2. 重启服务。<br>**预期**：奖期 Topic 走“0-1秒”重置逻辑；票据 Topic 走正常续传逻辑。互不干扰。 | 验证修改单一 Topic（如奖期）的 Redis 时间触发严重故障逻辑时，不影响其他 Topic（如票据）的正常续传。 |
| 6 | 异常机制 | 数据库连接失败重试 | **DB重试**：配置错误 IP，验证日志是否按配置间隔（Errorsecond）持续重试 | **预期**：日志出现连接错误，并每隔 1 分钟（配置值）重试一次，直到手动改回正确 IP 后恢复正常。 |
|  |  | Kafka 连接失败重试 | 模拟 Kafka 集群不可用（改配置或断网）。 | **预期**：服务不退出，日志持续打印重试信息，恢复后自动重连并继续推送。 |

前置：带宽限制2M

### 性能测试

| 序号 | 测试项 | 测试点 | 备注 |
| --- | --- | --- | --- |
| 1 | 基准容量测试 | 验证在 **2Mbps 带宽限制**下，单链路推送 50 万张票的耗时（预期 15 分钟内）及 TPS 稳定性。 | 关注带宽满载时的网络 I/O 表现。 |
| 2 | 正常高并发测试 | 验证在 **5倍线上票量**（150万张票+15万张中奖票）下，**仅运行 Kafka 推送**时的处理能力，重点验证能否能完成推送及 TPS 稳定性。 | **性能极限验证**：<br>验证Kafka 单链路在带宽瓶颈下的最大吞吐能力。 |
| 3 | 混合高并发测试 | 验证在 **5倍线上票量** 下，Kafka 与 SFTP **并发运行**导致带宽争抢时，SFTP 的连接/传输稳定性及核心业务（售票/开奖）的响应时间。 | 模拟资源耗尽的最恶劣场景，确保核心业务。 |
| 4 | 中断恢复压力测试 | 验证跨天严重中断恢复后，**第二轮轮询**（抓取从 `00:00:01` 至 `当前` 的全量积压数据）瞬间的 DB 负载稳定性及数据抓取完整性。 | **恢复能力验证**：<br>验证恢复模式切换回正常模式瞬间的抗压能力。 |
| 5 | 极限容量与临界值探测测试 | 服务`中断跨天`，单次需追平 `600万`张极端积压数据的场景（`4倍`世界杯期间积压票量） | 验证当前机制在长时间持续推送中的稳定性。 |
