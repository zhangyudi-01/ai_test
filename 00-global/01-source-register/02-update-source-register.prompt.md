# 提示词：更新上下文源注册表

## 任务说明

当有新资料补充或现有资料更新时，更新上下文源注册表。

## 触发场景

### 场景1：新增资料文件
- 在 `00-source-materials` 目录下新增了文件
- 需要将新文件添加到注册表

### 场景2：补充缺失资料
- 原本标注为"待补充"的资料现已提供
- 需要更新注册表中的状态和路径

### 场景3：资料版本更新
- 已有资料发布了新版本
- 需要更新版本/日期字段

### 场景4：AI生成资料
- AI生成了业务模块文档等资料
- 需要更新注册表状态

## 输入资料

1. **当前注册表**：`context-source-register.md`
2. **新增/更新的资料信息**：
   - 文件路径
   - 文件名称
   - 版本/日期
   - 资料类型

## 更新操作指引

### 操作1：添加新资料条目

为新增的资料添加注册表条目：

**步骤**：
1. 分配新的 Source ID（递增编号）
2. 确定资料类型和权威级别
3. 填写完整的表格行
4. 按 Source ID 顺序插入到注册表

**示例**：
```markdown
| **SRC-019** | 新资料类型 | 文件路径 | 资料名称 | v1.0 / 2026-06-10 | 中 | 使用范围 | 禁止用途 | 备注 |
```

### 操作2：更新缺失资料状态

将"待补充"或"待生成"的资料更新为实际路径：

**更新前**：
```markdown
| **SRC-013** | API文档 | 00-global/00-source-materials/05-api-docs/ | API接口文档 | 待补充 | **高** | 接口测试设计依据 | - | **待开发补充** |
```

**更新后**：
```markdown
| **SRC-013** | API文档 | 00-global/00-source-materials/05-api-docs/api-spec-v1.0.md | API接口文档 | v1.0 / 2026-06-10 | **高** | 接口测试设计依据 | - | 已补充 |
```

### 操作3：更新资料版本

更新已有资料的版本信息：

**更新字段**：
- 文件路径（如文件名变更）
- 版本/日期
- 备注（如有重大变更需说明）

**示例**：
```markdown
| **SRC-001** | 当前需求 | 00-global/00-source-materials/01-current-requirement/REQ-当前迭代需求-v2.md | 当前迭代需求文档 | v2 | **高** | 本次测试的主要依据 | 不允许被历史资料覆盖 | v2更新：新增XX功能 |
```

## 同步更新缺口清单

更新注册表后，同步更新 `context-source-gap-list.md`：

### 如果填补了缺口
1. 将对应缺口的状态改为"✅已补充"
2. 在备注中记录补充日期

### 如果新增了缺口
1. 在缺口清单中添加新的缺口条目
2. 分析影响范围和优先级

## 验证检查清单

更新后检查：
- [ ] Source ID 连续且无重复
- [ ] 文件路径准确且文件存在
- [ ] 权威级别符合规则
- [ ] 使用范围和禁止用途明确
- [ ] 待补充资料已明确标注
- [ ] 缺口清单已同步更新

---

**现在，请根据以上要求，更新上下文源注册表。**
