# 一、这个方案的起点：不是为了做工具，而是为了修 AI 的缺陷

AI 会理解偏，那就加人工逻辑灌输；  
AI 会越界，那就加边界约束；  
AI 产物不稳定，那就加审核卡点；

# 三、这个方案和 Open Spec 的关系

你问过一个很关键的问题：这个东西是不是本质上就是把 Open Spec 流程自动化？

我的判断是：

**它不是简单复制 Open Spec，而是站在测试视角，对 Open Spec 做了一层质量增强。**

Open Spec / SDD 的核心流程大概是：

需求输入  
→ AI 理解需求  
→ 拆成规格 / 原子任务  
→ 归档  
→ 后续开发、测试、实现都依赖这份归档内容

你发现的问题是：  
如果第一步 AI 理解需求时漏了内容，那么后面所有东西都会基于一份“有缺陷的归档”继续往下走。

这就是单点故障。

所以你这个方案比普通 Open Spec 多了几个关键东西：

1. **不允许 AI 一次性理解后直接归档。**
2. **AI 拆解完后，必须有人审核。**
3. **AI 拆解结果还要用另一个 AI 反向找茬。**
4. **归档内容不能只是摘要，必须是原子需求 + 模块归属 + 上下文关系 + 测试关注点。**
5. **后续生成测试方案和用例，也不能全量一次性输出，必须按模块拆。**

所以它更准确的定位是：

> **Open Spec 思想 + QA 强审核机制 + 动态提示词机制 + 分模块测试资产生成机制。**

这就比“自动化 Open Spec”更有价值。

# 四、你们已经讨论出的核心流程

你和 Gemini 后面已经碰撞出了几个非常重要的点，我把它们串起来。

## 1. 原始需求输入

输入的是原始需求文档。

这里一定要注意：**原始需求永远不能丢。**

不能因为 AI 生成了一个摘要、一个拆解表、一个测试方案，就把原始需求放一边。因为所有 AI 产物都可能有损失，原文才是最终依据。

所以第一份资产应该是：

```
01-原始需求文档.md
```

或者：

```
01-原始需求文档.docx / pdf
```

如果后面在编辑器里试跑，建议统一转成 Markdown，方便 AI 检索、引用、拆分。

---

## 2. 提示词先行

这是你在后面补充出来的关键点，而且非常重要。

你说：**所有让 AI 分析或产出的步骤，都应该先让 AI 输出一份专业提示词；人工审核提示词后，再把提示词喂给 AI 生成产物。**

这个点非常对。

因为提示词本身就是“下一步产物的方向盘”。

如果提示词写偏了，后面产物一定偏。

所以这个流程里不能只有：

```
需求 → AI → 产物
```

而应该变成：

```
需求 + 当前目标 → AI 生成专业提示词 → 人工审核提示词 → AI 根据审核后的提示词生成产物
```

这意味着每一步都有两个动作：

第一步：生成提示词。  
第二步：执行提示词。

比如：

生成原子需求之前，要先生成“原子需求拆解提示词”。

生成模块逻辑之前，要先生成“模块逻辑梳理提示词”。

生成测试方案之前，要先生成“测试方案生成提示词”。

生成测试用例之前，要先生成“测试用例生成提示词”。

生成自动化脚本之前，要先生成“自动化脚本生成提示词”。

这就是你说的“提示词每个过程中都得重新生成，并且需要人工审核”。

这个机制非常重要，因为每个模块、每次迭代、每个业务点的特殊逻辑都不一样，不能靠一套固定提示词打天下。

---

## 3. 原子需求拆解

这一步是整个流程的第一道大闸。

AI 要把原始需求拆成一条条原子需求。

但这里不能只是简单总结，比如：

```
用户可以报名赛事
用户可以查看积分
用户可以取消报名
```

这种太粗。

真正需要的是类似这样的结构：

```
需求编号：REQ-001
所属模块：赛事报名
原始需求引用：第 X 段 / 原文片段
触发条件：用户进入赛事详情页并点击报名
前置条件：用户已登录、赛事未截止、用户满足报名资格
业务规则：同一用户同一赛事只能报名一次
异常规则：不满足资格时提示原因
数据影响：写入报名记录，更新报名人数
上下游影响：影响积分、资格校验、赛事名单
测试关注点：重复报名、报名截止、资格不足、并发报名
待确认问题：需求未说明报名失败是否记录日志
```

也就是说，原子需求不是“短句摘要”，而是带结构、带来源、带测试价值的拆解结果。

这一步要解决的是：**防止 AI 把需求有损压缩。**

---

## 4. 双 AI 审核原子需求

你们讨论过“要不要用另一个 AI 再核对一次”。

我的判断是：**要用，但不能代替人工。**

这一步应该是：

AI 1：负责正向拆解需求。  
AI 2：负责反向审计需求。

AI 2 不应该重新写一份完整拆解，而是要专门找问题：

```
有没有遗漏原文中的条件？
有没有遗漏异常场景？
有没有遗漏跨模块影响？
有没有把原文没有的东西脑补进去？
有没有把多个需求合并得太粗？
有没有没有标注待确认问题？
有没有漏掉数据变更、权限、状态、时间、并发、接口影响？
```

最后产物应该是：

```
02-原子需求拆解表.md
03-原子需求审计报告.md
04-人工修正后的原子需求基线.md
```

这里最重要的是第三份：**人工修正后的原子需求基线。**

这份才可以作为后续流程的依据。

---

## 5. 人工审核原子需求

这一步不能省。

你之前已经判断得很准确：Open Spec 或类似流程最危险的地方，就是第一份归档内容如果不完整，后面所有内容都会跟着错。

所以这里应该设置一个明确规则：

> 原子需求没有人工审核通过，不允许进入后续测试方案生成。

人工审核要看什么？

重点不是看文字漂不漂亮，而是看有没有漏：

1. 正向流程有没有完整。
2. 异常流程有没有单独列出。
3. 边界值有没有体现。
4. 权限角色有没有体现。
5. 状态变化有没有体现。
6. 时间条件有没有体现。
7. 数据落库 / 数据变更有没有体现。
8. 上下游模块有没有体现。
9. 旧逻辑影响有没有体现。
10. 待确认问题有没有明确列出来。
11. AI 有没有脑补需求外的内容。
12. 原始需求中的每个关键句子是否都能追溯到某条原子需求。

最后这一步应该形成一个“可追溯”的基线文件。

# 五、逻辑灌输到底应该放在哪里？

这是你们讨论里最重要、但之前确实没有完全串好的地方。

你问：逻辑投喂应该在什么时候做？

是拆解完原子需求后，给 AI 喂某个模块需求，再把模块逻辑和上下文讲清楚？  
还是让 AI 自己先生成一份它理解的逻辑，然后你去改？

我的结论是：

> **逻辑灌输应该分两层做：全局逻辑先建底座，模块逻辑按模块单独修正。**

不能只做一次。

---

## 第一层：全局业务逻辑底座

这个在项目开始时就要准备。

它不是针对某一个需求，而是你们赛事运营项目长期稳定存在的业务常识。

比如：

```
赛事运营项目背景
核心业务模块
核心业务链路
通用状态流转
通用权限规则
常见数据表 / 字段
常见接口
常见异常场景
历史高频问题
测试重点偏好
AI 禁止脑补的边界
```

这份东西可以叫：

```
00-赛事运营项目全局上下文.md
```

它的作用是让 AI 不至于每次都从零理解项目。

但要注意，它只能作为背景，不能替代本次需求。

---

## 第二层：模块级逻辑灌输

这个才是你们重点讨论的“给 AI 讲逻辑”。

它应该发生在：

> **原子需求人工审核完成之后，生成测试方案之前。**

为什么是这个时机？

因为原子需求审核完成后，说明“要测什么”已经基本准确了。  
但生成测试方案之前，还需要让 AI 理解“这些需求背后的业务逻辑、上下文关系、隐藏影响”。

这一步最好采用：

> **AI 先生成它理解的模块逻辑 → 人工修正 → 形成模块逻辑基线。**

而不是完全由人从零写。

原因很简单：  
如果全部靠你写，那 AI 工具就变成体力活了。  
让 AI 先写一版，你只需要改错、补充、删脑补内容，效率更高。

模块逻辑基线应该包含：

```
模块名称
本次迭代涉及的原子需求
模块核心业务流程
前置条件
状态流转
上下游模块影响
数据影响
接口影响
权限影响
时间规则
异常场景
并发 / 幂等 / 重复提交
历史兼容逻辑
本次特殊逻辑
测试重点
AI 禁止扩展的边界
待确认问题
```

这一步非常关键。

因为它解决的是：**AI 只看字面需求、不懂隐藏逻辑的问题。**

# 六、为什么必须分模块生成测试方案和用例？

你最后补充的这个点，是整个方案从“理论能用”走向“实际能落地”的关键。

你说：你们现在的用例库本来就是按功能模块创建不同 Excel 文档，如果一次性让 AI 输出所有模块方案和用例，还是会受到长上下文限制。

这个判断完全正确。

所以后续不能是：

```
所有原子需求 + 所有模块逻辑 → 一次性生成完整测试方案 → 一次性生成所有用例
```

这样会重新掉进长上下文陷阱。

应该是：

```
全局需求拆解完成
→ 按模块拆分
→ 每个模块单独生成测试方案
→ 每个模块单独审核
→ 每个模块单独生成用例
→ 每个模块单独输出 Excel
```

也就是说，后半段必须是一个模块循环。

例如：

```
模块A：赛事报名
模块B：赛事审核
模块C：赛事变更
模块D：告警处理
模块E：任务解除
模块F：备注 / 预警 / 变更公告
```

每个模块独立跑一遍：

```
模块需求 → 模块逻辑 → 模块测试方案 → 模块测试用例 → 模块Excel
```

这样每次给 AI 的上下文都比较小，它的注意力更集中，产物质量也更可控。

# 七、测试方案这一层不能省

你们讨论到“输出用例之前还需要输出测试方案”，这个点也非常重要。

很多人用 AI 写测试用例时，会直接：

```
需求 → 测试用例
```

这样很容易漏。

因为测试用例是细节层，测试方案是战略层。

如果测试方案没有先想清楚，那么用例再多，也可能方向错。

所以应该是：

```
需求拆解 → 模块逻辑 → 测试方案 → 测试用例
```

测试方案里应该先列出：

```
功能测试范围
不测范围
核心业务链路
模块测试点
异常测试点
边界测试点
权限测试点
数据一致性测试点
接口测试点
兼容旧逻辑测试点
并发 / 幂等测试点
回归影响范围
风险点
待确认问题
```

然后再根据测试方案生成测试用例。

这样做的好处是：

测试方案先保证“测的方向对”。  
测试用例再保证“每个点测得细”。

# 八、测试方案也需要双 AI 审核

你问测试方案是不是也需要双 AI 审核。

我的判断是：**需要。**

因为测试方案一旦漏了，后面的用例必然漏。

测试方案阶段应该这样：

AI 1：根据模块原子需求 + 模块逻辑基线，生成模块测试方案。  
AI 2：拿着同样的输入，对 AI 1 的测试方案做审计。  
人工：看 AI 2 的意见，修正测试方案，形成最终版。

AI 2 要审什么？

```
是否覆盖所有原子需求？
是否覆盖人工补充的模块逻辑？
是否漏掉异常流程？
是否漏掉边界值？
是否漏掉上下游影响？
是否漏掉数据校验？
是否漏掉权限角色？
是否漏掉状态流转？
是否漏掉历史兼容？
是否漏掉并发、幂等、重复提交？
是否存在需求外扩展？
是否存在测试范围过大或过窄？
```

最后产物是：

```
模块A-测试方案-v1.md
模块A-测试方案审计报告.md
模块A-测试方案-人工确认版.md
```

# 九、测试用例生成也要按模块、按测试点继续拆

就算测试方案确认了，也不建议让 AI 一次性生成整个模块所有用例，特别是模块比较大时。

更稳的方式是：

```
模块测试方案
→ 按测试点分组
→ 每组测试点单独生成用例
→ 每组用例单独审核
→ 最后合并成模块 Excel
```

比如模块 A 的测试方案里有这些测试点：

```
A1 正常报名
A2 报名资格校验
A3 重复报名
A4 报名截止时间
A5 并发报名
A6 报名后数据落库
A7 报名失败提示
A8 取消报名影响
```

那就不要一次性生成 A1-A8 的所有用例。

可以分批：

```
第1批：A1 + A2
第2批：A3 + A4
第3批：A5
第4批：A6 + A7 + A8
```

尤其是并发、数据一致性、上下游影响这种复杂点，应该单独生成，单独审核。

这样更符合你说的“拆得再细一点”。

# 十一、这个流程里的几个核心原则

## 原则一：AI 不能直接产最终物

任何最终产物都不能是 AI 直接输出就算完成。

必须经过：

```
AI 产出 → AI 审计 → 人工审核 → 人工确认版
```

---

## 原则二：提示词也是产物，也要审核

你这个点非常重要。

提示词不是随便写的“命令”，它本身就是流程资产。

所以提示词也要有版本：

```
测试方案提示词-AI版
测试方案提示词-人工修正版
测试方案提示词-最终执行版
```

否则你后面复盘时根本不知道为什么这次 AI 产物质量好，下一次质量差。

---

## 原则三：原始需求必须可追溯

每条原子需求、每个测试点、每条用例，最好都能追溯到原始需求。

比如用例里加字段：

```
需求编号
原子需求编号
测试方案编号
原文引用
```

这样后面别人质疑“你这个用例从哪来的”，你能说清楚。

---

## 原则四：人工不是做体力活，而是做关键判断

这个流程不是让人把 AI 产物从头改到尾。

真正理想的状态是：

AI 做初稿。  
AI 做审计。  
人只在关键位置判断方向、补隐藏逻辑、删脑补内容、确认最终版。

你在这个方案里的价值不是“操作工具的人”，而是“质量卡点的人”。

---

## 原则五：拆得越细，AI 越准

这个是对抗长上下文缺陷的关键。

不要让 AI 一次性处理：

```
全部需求
全部模块
全部测试方案
全部测试用例
```

而是要拆成：

```
一个模块
一组测试点
一批用例
一个输出文件
```

这和你们现有“按模块维护 Excel 用例库”的方式天然一致。

# 十二、目前方案里还要补齐的几个点

虽然整体方向已经很好了，但我觉得还有几个地方需要补上，不然后面试跑时会卡。

## 1. 缺少“测试范围边界”定义

AI 很容易扩展需求之外的东西。

所以每个模块都应该有：

```
本次要测
本次不测
允许合理补充
禁止扩展范围
需要人工确认后才能加入的内容
```

尤其你之前也说过，AI 改代码时会按自己的理解乱加需求之外的东西。生成测试方案和用例也是一样。

---

## 2. 缺少“待确认问题清单”

AI 在分析需求时，遇到不明确的地方不能脑补。

应该强制它输出：

```
待确认问题
影响范围
如果不确认，可能导致哪些测试遗漏
建议找谁确认：产品 / 开发 / 测试 / 运营
```

这个清单很有价值，因为它能反向暴露需求文档的问题。

---

## 3. 缺少“用例质量标准”

否则你不知道 AI 输出的用例算不算好。

建议先定义一份用例质量标准，比如：

```
是否覆盖原子需求
是否覆盖异常流程
是否覆盖边界值
是否包含前置条件
是否包含操作步骤
是否包含预期结果
是否可执行
是否避免重复
是否避免需求外扩展
是否标注优先级
是否标注关联模块
是否标注数据准备
```

之后 AI 2 审计用例时，就按照这份标准审。

---

## 4. 缺少“历史用例库对齐”

你说你们现在用例库是按模块 Excel 管理的。

那后面试跑时，不应该完全从零生成。

应该让 AI 参考历史同模块用例结构，例如：

```
历史模块 Excel 字段
历史用例命名风格
优先级规则
用例颗粒度
常见前置条件写法
预期结果写法
```

这样生成的用例更容易融入你们现有用例库。

---

## 5. 缺少“产物对比复盘”

为了证明这个流程有效，后面试跑时应该做对比。

比如拿同一份需求，做三组：

```
A组：直接让 AI 生成用例
B组：按功能点喂给 AI 生成用例
C组：按你这个完整流程生成用例
```

然后对比：

```
遗漏需求数量
遗漏异常场景数量
人工修改耗时
最终可用用例数量
用例重复率
需求外扩展数量
待确认问题发现数量
```

这样你后面给老板或团队展示时，不只是说“我觉得效果好”，而是有证据。

# 十五、最终结论

你和 Gemini 讨论出来的方案，完整串起来后应该是这样的：

> 先用 Open Spec 的思想做需求原子化拆解，但不盲信 AI；  
> 每一步 AI 执行前，都先动态生成专业提示词，并由人工审核；  
> 原子需求拆解后，用第二个 AI 反向审计，再由人确认；  
> 进入模块后，不直接生成用例，而是先让 AI 生成模块逻辑理解，再由人补充隐藏逻辑和上下文关系；  
> 然后按模块生成测试方案，测试方案也要双 AI 审核和人工确认；  
> 最后再按模块、按测试点分批生成测试用例，并继续审计；  
> 所有产物都要保留原文追溯、提示词版本、AI 审计记录和人工确认版。

这套东西的核心不是“AI 自动干活”，而是：

**让 AI 在可控边界内干活，让人只在最关键的位置做判断。**

你现在先在编辑器里验证这个流程，是最正确的路线。先别做页面，先别做复杂工具。只要这个流程能在一个真实赛事运营需求上跑通，并且最终用例确实比普通 AI 生成质量高，那它后面再工具化就很自然了。

# 一、第二个 AI 到底什么时候接入？

不要理解成只有一个审核点。

它应该分层接入。

但一开始试跑时，也不要搞得太重。我们先分成 **必须接入** 和 **可选接入**。

---

## 1. 原子需求拆解后：必须接入

这是第一个最重要的节点。

流程是：

```
原始需求
→ AI 1 生成原子需求拆解表
→ AI 2 审核原子需求是否遗漏、误解、脑补
→ 人工终审
→ 形成原子需求基线
```

为什么这里必须接入？

因为原子需求是后面所有东西的源头。

如果这里漏了，后面测试方案、测试用例、自动化脚本都会跟着漏。

所以 AI 2 在这里的任务不是重新拆一遍，而是专门找问题：

```
AI 1 有没有漏掉原始需求里的条件？
有没有把多个需求合并得太粗？
有没有漏异常流？
有没有漏边界值？
有没有漏权限、状态、时间、数据影响？
有没有把原文没有的东西脑补进去？
有没有遗漏待确认问题？
```

这个节点的 AI 2 非常有价值。

---

## 2. 模块逻辑理解后：建议接入，但不能完全依赖它

这个节点比较特殊。

流程是：

```
模块需求包
→ AI 1 生成模块逻辑理解
→ AI 2 审核模块逻辑是否和需求一致
→ 人工补充隐藏逻辑、上下文关系
→ 形成模块逻辑基线
```

这里 AI 2 可以帮你检查：

```
AI 1 有没有理解错模块流程？
有没有漏掉上下游影响？
有没有把需求外的逻辑加进去？
有没有状态流转不完整？
有没有数据影响没说清楚？
```

但是注意：**模块隐藏逻辑、历史坑、业务潜规则，AI 2 很可能也不知道。**

所以这个节点不能指望 AI 2 完全兜底。

这里真正的核心还是你人工补充。

AI 2 的作用是帮你先扫一遍明显问题，减少你审核压力。

---

## 3. 测试方案产出后：必须接入

这个就是你刚才问的：“是不是等测试方案出来再审核？”

答案是：**是的，测试方案出来后必须让 AI 2 审核。**

流程：

```
模块需求包 + 模块逻辑基线
→ AI 1 生成测试方案
→ AI 2 审核测试方案
→ 人工确认测试方案
→ 进入用例生成
```

这里非常关键。

因为测试方案是战略层。

测试方案如果漏了某个测试方向，后面的用例就算写得再细，也不会覆盖那个方向。

AI 2 在这里要审：

```
测试方案是否覆盖所有原子需求？
是否覆盖人工补充的模块逻辑？
是否覆盖异常流程？
是否覆盖边界值？
是否覆盖权限、状态、时间、数据、接口影响？
是否覆盖上下游模块？
是否覆盖历史兼容和回归范围？
是否存在需求外扩展？
是否有待确认问题没有列出来？
```

这个节点必须接。

---

## 4. 测试用例产出后：必须接入

流程：

```
确认后的测试方案
→ AI 1 生成测试用例
→ AI 2 审核测试用例
→ 人工终审
→ 形成最终 Excel / 用例库
```

这里 AI 2 要审的是执行层质量：

```
用例是否覆盖测试方案中的每个测试点？
是否有测试点只写了标题但没有具体步骤？
前置条件是否清楚？
测试数据是否明确？
操作步骤是否可执行？
预期结果是否可验证？
有没有重复用例？
有没有漏掉反向场景？
有没有需求外扩展？
优先级是否合理？
是否能落到你们现有 Excel 用例库结构里？
```

这个节点也必须接。

---

## 5. 提示词生成后：关键步骤建议接入

你前面说过一个非常重要的点：**每个 AI 执行步骤之前，都应该先生成专业提示词，然后人工审核提示词。**

那第二个 AI 要不要审核提示词？

我的建议是：

**关键步骤的提示词要让 AI 2 审核，普通步骤可以人工审核即可。**

关键步骤包括：

```
原子需求拆解提示词
测试方案生成提示词
测试用例生成提示词
自动化脚本生成提示词
```

AI 2 审核提示词时，不是看文采，而是看它有没有把任务约束讲清楚：

```
是否明确输入资料？
是否明确输出格式？
是否明确禁止脑补？
是否要求标注原文来源？
是否要求列出待确认问题？
是否要求覆盖异常流、边界值、权限、状态、数据影响？
是否定义了本次测试范围和不测范围？
是否要求不要生成需求外内容？
```

但是这个节点一开始不建议做得太复杂。

第一版编辑器试跑时，可以先让你自己审核提示词。  
等流程顺了，再加 AI 2 审核提示词。

---

# 二、第二个 AI 的资料从哪里来？

这是核心。

**第二个 AI 不能只看第一个 AI 的产物。**

如果它只看 AI 1 的结果，它很容易被 AI 1 带偏。

比如 AI 1 原子需求漏了一个条件，AI 2 如果只看 AI 1 的拆解表，它根本不知道漏了什么。

所以 AI 2 必须拿到“审核包”。

---

# 三、什么是审核包？

审核包就是每次让 AI 2 审核时，给它的一组资料。

它至少包含四类东西：

```
1. 原始依据
2. AI 1 使用的提示词
3. AI 1 输出的产物
4. 审核标准
```

也就是说，AI 2 不是凭感觉审核，而是基于资料审。

---

## 原子需求审核包

当 AI 2 审核原子需求拆解时，它应该拿到：

```
1. 原始需求文档
2. 项目全局上下文
3. 原子需求拆解提示词最终版
4. AI 1 输出的原子需求拆解表
5. 原子需求审核标准
```

它要做的是：

```
拿原始需求逐条对比 AI 1 的原子需求拆解表。
找遗漏、误解、合并过粗、需求外脑补、待确认问题缺失。
```

所以这里的资料来源主要是：

```
原始需求文件 + AI 1 提示词 + AI 1 产物 + 审核标准文件
```

你不需要重新人工投喂一遍。  
你只需要把这些文件整理好，让第二个 AI 能拿到。

---

## 模块逻辑审核包

当 AI 2 审核模块逻辑时，它应该拿到：

```
1. 项目全局上下文
2. 当前模块需求包
3. 当前模块相关原子需求
4. 模块逻辑梳理提示词最终版
5. AI 1 输出的模块逻辑理解
6. 模块逻辑审核标准
```

它要检查：

```
模块流程是否对
上下游关系是否漏
状态流转是否完整
数据影响是否完整
有没有需求外脑补
有没有明显和原始需求冲突
```

但是这一阶段你仍然要补充人脑逻辑。

因为 AI 2 也只能基于资料看问题。

如果某个隐藏逻辑资料里没有写，AI 2 不一定能发现。

---

## 测试方案审核包

当 AI 2 审核测试方案时，它应该拿到：

```
1. 项目全局上下文
2. 当前模块需求包
3. 当前模块原子需求基线
4. 当前模块逻辑基线
5. 测试方案生成提示词最终版
6. AI 1 输出的测试方案
7. 测试方案审核标准
```

这里最关键的是：

**AI 2 必须拿到“模块逻辑基线”。**

因为测试方案不是只覆盖字面需求，还要覆盖你人工补充过的上下文逻辑。

比如你补充了：

```
这个模块涉及状态回退
这个模块影响告警数量
这个模块和任务解除有关
这个模块需要关注数据重复写入
```

那 AI 2 审核测试方案时，就要专门检查 AI 1 有没有把这些点写进方案。

---

## 测试用例审核包

当 AI 2 审核测试用例时，它应该拿到：

```
1. 当前模块需求包
2. 当前模块原子需求基线
3. 当前模块逻辑基线
4. 当前模块测试方案最终版
5. 测试用例生成提示词最终版
6. AI 1 输出的测试用例
7. 用例质量标准
8. 你们现有 Excel 用例模板
```

这里 AI 2 要检查的是：

```
用例是否覆盖测试方案
用例是否可执行
步骤是否清楚
预期结果是否明确
是否遗漏反向场景
是否遗漏边界值
是否重复
是否不符合你们 Excel 用例库格式
```

# 五、第二个 AI 每次应该拿“同一份资料”还是“更多资料”？

这个要分情况。

## 原则一：AI 2 必须拿到 AI 1 拿过的核心资料

否则它没法判断 AI 1 有没有漏。

比如 AI 1 生成测试方案时拿到了：

```
模块需求包
模块逻辑基线
测试方案提示词
```

那 AI 2 审核测试方案时，也必须拿到这些。

---

## 原则二：AI 2 要额外拿到 AI 1 的产物

这是它要审核的对象。

比如：

```
AI 1 测试方案
AI 1 测试用例
AI 1 原子需求拆解表
```

---

## 原则三：AI 2 要额外拿到审核标准

AI 1 是生成型任务。  
AI 2 是审计型任务。

所以 AI 2 需要额外拿审核标准。

比如：

```
原子需求审核标准
测试方案审核标准
测试用例审核标准
```

这三个标准最好提前沉淀成固定文件。

# 六、第二个 AI 不能做什么？

这个也要提前限制。

第二个 AI 不是来重写所有东西的。

它不能一上来直接给你一份新版测试方案，这样会把流程搞乱。

它应该只输出审计结果。

格式可以固定成：

```
问题编号
问题类型：遗漏 / 误解 / 脑补 / 颗粒度过粗 / 边界缺失 / 上下文缺失
对应原始依据
当前 AI 1 产物位置
问题说明
修改建议
严重级别：高 / 中 / 低
是否必须人工确认
```

这样你人工审核时很清楚。

否则 AI 2 又长篇大论输出一大堆，你会更累。

我建议你先按这个最小闭环跑：

```

→ 人工确认测试方案
→ AI 1 生成测试用例
→ AI 2 审测试用例

```

```
→ 人工确认提示词
→ AI 1 执行
→ AI 2 审核产物
→ 人工确认产物
```

这样不会一开始就太重。

# 二、角色定义

为了后面执行不乱，先把角色定清楚。

## 1. AI-P：提示词生成 AI

职责：

> 专门根据当前任务、输入资料、输出目标，生成一份专业提示词。

它不直接生成测试方案，也不直接生成测试用例。

它只负责：

例如：

```
原子需求拆解提示词
模块逻辑梳理提示词
测试方案生成提示词
测试用例生成提示词
测试用例审核提示词
```

```
需求整体理解报告
原子需求拆解表
模块逻辑理解报告
测试方案
测试用例
用例 Excel 初稿
自动化脚本初稿
```

## 3. AI-2：审核 AI / 审计 AI

职责：

> 不负责直接生成最终产物，只负责审核 AI-P 的提示词和 AI-1 的产物。

AI-2 的角色要固定成：

```
极其挑剔的资深测试审计员
需求一致性检查员
上下文遗漏检查员
需求外扩展拦截员
```

AI-2 的输出不应该是“我帮你重写一版”，而应该是：

```
问题清单
遗漏点
误解点
脑补点
风险点
修改建议
是否必须人工确认
```

你的职责是：

你是最后一道质量闸门。

# 三、必须建立的固定规则

## 规则 1：凡是 AI 要分析或产出，前面必须先生成提示词

不能直接：

```
需求 → AI → 产物
```

必须是：

```
当前资料
→ AI-P 生成提示词草稿
→ AI-2 审核提示词
→ 人工确认提示词
→ AI-1 根据最终提示词生成产物
→ AI-2 审核产物
→ 人工确认产物
```

这是全流程的固定模式。

---

## 规则 2：AI-2 审核提示词的时机

AI-2 必须审核这些提示词：

```
1. 需求整体理解提示词
2. 原子需求拆解提示词
3. 原子需求审核提示词
4. 模块拆分提示词
5. 模块逻辑梳理提示词
6. 模块逻辑审核提示词
7. 测试范围边界梳理提示词
8. 测试方案生成提示词
9. 测试方案审核提示词
10. 测试点拆分提示词
11. 测试用例生成提示词
12. 测试用例审核提示词
13. 用例格式化 / Excel 转换提示词
14. 最终全局一致性审核提示词
15. 自动化脚本生成提示词，如果后续做脚本
16. 自动化脚本审核提示词，如果后续做脚本
```

注意一点：

**AI-2 自己执行审核时，也需要审核提示词，但不能让 AI-2 无限审核自己的审核提示词。**

所以这里定一个边界：

> AI-2 的审核提示词由 AI-P 生成，再由 AI-2 按“提示词质量标准”审核一次，最后人工确认。  
> 人工确认后，不再继续让 AI-2 审核自己的审核提示词，避免无限循环。

---

## 规则 3：任何最终产物都必须有人类确认版

每个产物至少有三个版本：

```
AI 初稿
AI-2 审核报告
人工确认版
```

最终只能使用人工确认版。

例如：

```
atomic-requirements.ai.md
atomic-requirements.audit.md
atomic-requirements.final.md
```

后续流程只能引用：

```
atomic-requirements.final.md
```

不能引用 AI 初稿。

---

## 规则 4：所有内容必须可追溯

每条原子需求、每个测试点、每条用例，都要能追溯到原始需求。

建议所有产物都带这些字段：

```
原始需求位置
原文摘录
原子需求编号
所属模块
测试方案编号
测试点编号
用例编号
```

这样后面别人问“这个用例从哪来的”，你能直接反查。

---

## 规则 5：必须按模块分治

不能一次性让 AI 输出所有模块的测试方案和所有用例。

正确流程是：

```
全局需求拆解
→ 按模块拆分
→ 每个模块单独生成逻辑理解
→ 每个模块单独生成测试方案
→ 每个模块单独生成测试用例
→ 每个模块单独输出 Excel
```

如果某个模块很大，还要继续拆成测试点批次。

# 五、全流程总览

最终流程是：

```
阶段 0：基础资产准备
阶段 1：原始需求接入与清洗
阶段 2：需求整体理解
阶段 3：原子需求拆解
阶段 4：AI-2 原子需求审计
阶段 5：人工确认原子需求基线
阶段 6：按模块拆分需求
阶段 7：进入单模块循环
    7.1 模块需求包确认
    7.2 模块逻辑理解
    7.3 AI-2 审核模块逻辑
    7.4 人工补充隐藏逻辑
    7.5 测试范围边界确认
    7.6 模块测试方案生成
    7.7 AI-2 审核测试方案
    7.8 人工确认测试方案
    7.9 测试点拆分
    7.10 按测试点批次生成用例
    7.11 AI-2 审核测试用例
    7.12 人工确认测试用例
    7.13 合并模块 Excel
阶段 8：跨模块一致性审核
阶段 9：最终归档与复盘
```

下面开始细拆。

# 阶段 7：模块逻辑理解与逻辑灌输

这是你最关注的地方。

逻辑灌输的正确时机是：

> **模块需求包确认后，测试方案生成前。**

不能太早，也不能太晚。

太早：原子需求还没确认，容易基于错误内容灌输。  
太晚：测试方案已经生成，再补逻辑就返工。

所以最佳时机就是：

```
模块需求包 final
→ AI 生成模块逻辑理解
→ AI-2 审核
→ 你人工补充隐藏逻辑
→ 形成模块逻辑基线
→ 再生成测试方案
```

# 最终一句话版

这个项目最终不是“AI 自动生成测试用例”，而是：
